# 第一次配置提供商

## 目标

把一个模型提供商配置到“可以稳定返回文本”的状态。  
首轮只需要成功，不需要一步到位把所有高级参数都填满。

## 建议你先准备好

- 一个可用的 API Key
- 你要接入的服务商地址
- 至少一个确认能用的模型名称

## 推荐的首配策略

### 最简单的做法

先配一个最常用、最稳定的模型提供商，只测文本聊天。  
如果这个阶段你还同时开了代理、自定义头、原始 JSON 请求体和各种额外参数，出问题时会很难判断是哪一层出错。

### 稍微进阶一点

当基础链路稳定后，再逐步利用这些能力：

- 自定义请求头
- 参数表达式
- 原始 JSON 请求体
- 提供商级代理设置
- 模型级别的特殊参数

## 配置时重点检查的字段

| 项目 | 说明 |
| --- | --- |
| Base URL | 是否是该服务实际要求的接口根地址 |
| API Key | 是否粘贴完整，是否有多余空格 |
| 模型名称 | 是否与服务端实际支持的模型标识一致 |
| 请求兼容性 | 是否属于 OpenAI / Anthropic / Gemini 或兼容格式 |
| 代理 | 若启用代理，确认协议、地址和鉴权信息都正确 |

## 一个稳妥的验证顺序

1. 保存提供商配置。
2. 只选择一个模型。
3. 回到聊天页发一句最普通的文本。
4. 看到正常返回后，再去尝试图片、语音、工具或更复杂的参数。

## 配好之后你可以继续做什么

- 配第二个模型，做模型切换
- 给不同能力单独选模型，比如聊天模型、TTS 模型、Daily Pulse 模型
- 如果你需要兼容接口，继续使用请求头覆盖和原始 JSON 请求体

## 常见问题

### 保存了但就是不工作

优先回头核查这三项：

- Base URL 是否写对
- API Key 是否有效
- 模型名是否真的存在

### 在手表上配置太费劲

这是正常的。  
更推荐在 iPhone 端配完，再让同步系统把配置带过去。
