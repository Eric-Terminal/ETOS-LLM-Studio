# 工具与 MCP

ETOS LLM Studio 的工具系统不是附属功能，而是产品结构里非常核心的一层。  
它的目标不是“挂几个 API 让模型调用”，而是把工具开关、审批、来源和会话级启用都真正做成可管理的系统。

## 工具分层

当前主要有三类能力：

- MCP 工具
- Shortcuts 工具
- 本地工具

这些能力会在工具中心里统一治理，而不是分散到不同角落。

## MCP 支持

项目内置完整 MCP 客户端链路，重点不只是“能连上”，还包括：

- 流式 HTTP / SSE 传输
- 重连与超时治理
- 握手治理
- 能力协商
- 会话级启用

如果你需要把外部服务、知识库或自动化流程接进聊天，这一层会非常关键。

## 审批策略

工具好不好用，不只看它是否存在，也看它执行时是否顺手。

### 你需要留意的点

- 可以控制工具是否默认启用
- 可以设置审批策略
- 不同会话可以有不同启用状态

这意味着你可以把一个会话保持成纯聊天，也可以让另一个会话变成“默认带工具的工作流”。

## 本地工具与文件能力

本地工具并不只是示意性质。

### 已覆盖的方向

- 搜索文件
- 分块读取
- 差异查看
- 局部编辑
- 移动、复制、删除
- SQLite 数据库操作
- 网页卡片展示
- 反馈工单自动提交

## 什么时候适合开启工具

### 推荐

- 你已经确认基础聊天可用
- 你明确知道这个会话为什么需要工具
- 你愿意为更强能力接受更复杂的链路

### 不推荐

- 还没配通模型时就一口气开很多工具
- 不清楚审批策略会带来什么效果
- 在手表上第一次探索时就把全部工具打开

## 一条实用经验

先把工具开关和审批策略理解清楚，再去接 MCP 服务器。  
否则一旦行为超出预期，你很难判断是服务端返回、客户端治理，还是会话配置本身造成的。
