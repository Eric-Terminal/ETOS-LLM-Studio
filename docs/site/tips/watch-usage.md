# Apple Watch 使用建议

ETOS LLM Studio 的手表端**不是附属遥控器**，但它**也不应该被当成"所有事情都在这里完成"的唯一入口**。最舒服的体验通常来自**明确的双端分工**。

::: tip 读这一页之前
建议你已经在 iPhone 上跑通了基础聊天，并配置了 [同步与备份](/modules/sync-backup) 让两端能互通。
:::

## Watch 端能做什么 / 不能做什么

### 能做（且做得很好）

| 任务 | 为什么适合 Watch |
| --- | --- |
| **快速发起对话** | 抬腕 → 数字表冠回到表盘 → 滑出 App → 直接发消息 |
| **语音输入聊天** | 比手机上更顺手——本来就靠麦克风用得最多 |
| **接收 Daily Pulse 推送** | 早上抬腕看一眼今天的卡片摘要 |
| **继续之前已有的会话** | 走路 / 排队 / 等车时回 AI 一句 |
| **碎片时间短轮聊天** | 一句话问题 → 一句话回答，闭环 |

### 不能做（或做起来很难）

| 任务 | 为什么不适合 Watch |
| --- | --- |
| 添加 / 编辑提供商 | 表单字段多，输入 API Key 极痛苦 |
| 填代理地址、用户名、密码 | 同上 |
| 导入世界书 / 大批量记忆 | 屏幕太小看不清条目 |
| 调显示系统 | 字体、背景、模糊度等需要在大屏上预览 |
| 管理 MCP 服务器 | URL、Bearer Token、OAuth 流程都得 iPhone 完成 |
| 管理工具审批策略 | 需要逐项审视，Watch 屏幕不够 |
| 反馈工单 | 截图、附日志都更适合在 iPhone 完成 |

## 为什么要这样分工

**不是因为 Watch 端能力不行**，而是因为：

- **手表输入成本更高**——文字输入费力，深层菜单更难导航
- **深层配置路径本来就更适合大屏幕**——表单字段一多就糟
- **手机更适合做批量管理和排障**——需要大屏 + 多窗口切换

## 推荐工作流

### 首次设置

1. 在 **iPhone** 上把提供商、模型和基础偏好配好
2. 打开 **设置 → 显示与体验 → 同步与备份** 启用 Apple Watch 同步
3. 等同步状态显示「**同步成功**」
4. 在 Watch 上把它当成**一个已经准备好的"随身交互端"**

### 日常使用

| 场景 | 设备 |
| --- | --- |
| 早上看 Daily Pulse 推送 | Watch 抬腕扫一眼 |
| 觉得某张卡片想深入 → 详细阅读 / 互动 | 转到 iPhone |
| 走路时突然想到问题 | Watch 语音输入 |
| 看到 AI 回答太长不方便读 | 让它在 iPhone 上继续，回家看 |
| 想给某条卡片点赞 / 隐藏 | iPhone（Watch 看通知就够了）|

## Watch 上具体能做的事

### 日常聊天

- **短消息来回**
- **已有会话继续聊**（同步过来的）
- **语音输入补充**（按表冠或抬腕说话）

### 主动提醒

- **查看 Daily Pulse 推送**
- 从通知动作**继续进入聊天**

### 轻量消费内容

- **读较短回答**
- **快速浏览重点**
- **在必要时转去手机**做更深操作

## 不建议做的事

- ❌ 第一次就在手表上**配置所有提供商**
- ❌ 把复杂的工具实验放在 **Watch 上做首轮测试**
- ❌ 在**没有同步稳态**前双端同时改很多配置（会产生冲突）
- ❌ 在 Watch 上**编辑长文本**——能用语音就用语音

## 同步不稳怎么办

参考 [同步与备份 → 进阶选项 → 同步出问题排查](/modules/sync-backup#%E5%90%8C%E6%AD%A5%E5%87%BA%E9%97%AE%E9%A2%98%E6%8E%92%E6%9F%A5)。

常见原因：

- 蓝牙断了 / 不在同 Wi-Fi → 检查连接
- 双端同时编辑 → 后写入覆盖前面 → **改成"只在 iPhone 配置"原则**
