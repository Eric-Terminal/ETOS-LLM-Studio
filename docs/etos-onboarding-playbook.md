# ETOS 新手教程设计手册

## 文档目的

这是一份完全自洽的内部手册，用来指导 ETOS LLM Studio 的新手教程设计与实现。

以后如果没有任何额外上下文，只看这份文档，也应该能直接回答下面几个问题：

- ETOS 的新手教程应该分成哪些层次；
- 哪些页面适合做教程，哪些不适合；
- 第一批教程应该先做什么；
- iOS 和 watchOS 应该怎么分别落地；
- 实现时要遵守哪些工程边界；
- 怎样判断一条教程是否真的“教会了”。

这份文档不是需求草稿，也不是灵感收集，而是以后默认执行的工作手册。

## 目标定义

ETOS 的新手教程不是一次性欢迎弹窗，而是一套能长期复用的学习系统。它至少要做到：

- 用户第一次打开时，知道最短上手路径；
- 用户中途卡住时，能快速回到教程中心补课；
- 用户不是只“看懂”，而是真的完成一次关键操作；
- 用户做完后能立刻知道自己成功了；
- iOS 和 watchOS 都有对应版本，但交互密度按平台分别适配；
- 教程本身不污染真实数据、不阻塞主流程、不制造额外负担。

如果一条教程只是介绍概念、没有帮助用户完成动作，那它只能算说明文，不算教程。

## 教程系统的四层结构

ETOS 以后默认采用四层结构，而不是只做一个入口或只做弹窗。

### 第一层：首次触达层

作用是把用户带去正确入口，而不是在首次启动时讲完所有内容。

适合放在这一层的内容：

- 首次配置提供商；
- 首次进入聊天时的最短提示；
- 首次接触工具箱时的职责说明；
- 首次启用 Daily Pulse 时的基础引导；
- 重大版本更新后的新能力提醒。

这一层必须满足：

- 触发条件明确；
- 文案短；
- 最好只做分流与提醒；
- 用户可以稍后再回来学完整内容。

这一层不适合做：

- 长篇说明；
- 多步骤记忆；
- 复杂演示；
- 一次性塞入全部功能介绍。

### 第二层：常驻教程中心

必须提供一个以后还能反复进入的统一入口。默认可以命名为：

- 新手教程；
- 使用提示；
- 上手中心。

这个入口要长期存在，不能依赖“第一次打开时刚好看过”。它至少应包含以下栏目：

- 快速开始；
- 提供商与模型；
- 聊天基础；
- 工具箱；
- 会话与 Daily Pulse；
- 进阶技巧；
- 新功能。

如果没有这一层，用户一旦错过首次提示，后续就只能靠猜。

### 第三层：场景内补充层

也就是直接挂在具体功能页里的说明入口，例如顶部说明卡、空态提示、首屏短提示。

它的职责不是完整教学，而是：

- 告诉用户这个页面是做什么的；
- 告诉用户最关键的开关或入口在哪；
- 告诉用户完整教程去哪里看。

ETOS 当前已经有一部分这类能力，例如多处设置页顶部的 `settingsIntroCard`。以后不是推翻它，而是把它纳入统一体系：

- 继续保留为轻量说明层；
- 需要动手练习的功能，再补状态清单页和实操页；
- 需要长期维护的主线，再补进常驻教程中心。

### 第四层：新功能回看层

新功能说明应该单独存在，但它不替代正式教程。

它负责三件事：

- 告诉用户新版本多了什么；
- 提供进入深度教程的入口；
- 允许用户以后回看。

新功能页不应该承担完整教学责任。它只负责“发现”和“再进入”。

## 教程页面的三种标准模板

ETOS 以后默认只需要三种核心模板，不要一开始设计复杂的教程框架。

### 模板一：状态清单页

适合场景：

- 首次配置提供商；
- 首次完成工具箱启用；
- 首次完成同步；
- 首次建立 Daily Pulse 使用闭环。

状态清单页必须具备：

- 明确的完成项；
- 实时状态显示；
- 每项都能直接跳到对应页面；
- 用户能一眼看出“还差哪一步”。

状态清单页应该回答：

- 已经完成了什么；
- 还缺什么；
- 现在该点哪里；
- 做完后会发生什么。

状态清单页不应该写成泛泛而谈的说明书。

### 模板二：实操练习页

适合场景：

- 第一次发送消息；
- 第一次切换模型；
- 第一次检查工具是否成功暴露；
- watch 端高频手势或轻量交互；
- 需要用户“自己做一遍”才算掌握的动作。

实操练习页必须遵守五条规则：

1. 一页只教一个动作。  
   不要同时教添加提供商、切换模型、发消息、调参数。

2. 使用真实交互。  
   能让用户自己点、滑、开关、发送，就不要退化成纯图文说明。

3. 使用安全数据。  
   优先用本地样例、临时状态、演示内容，不污染真实会话与真实配置。

4. 允许反复进入。  
   用户可以中途退出，也可以以后回来再练一次。

5. 成功反馈即时。  
   做对以后立刻显示“已完成”、对勾、toast 或状态变化。

如果一页不能让用户产生“我刚刚真的完成了这一步”的感觉，这页就不够好。

### 模板三：说明卡片页

适合场景：

- 解释总开关和单项开关的关系；
- 解释功能职责边界；
- 解释 iPhone 端和 watch 端分工；
- 解释限制条件和前置要求；
- 解释某个系统为什么存在、什么时候该用。

说明卡片页的规则：

- 文字必须服务于动作；
- 只解释用户此刻需要知道的内容；
- 必须能把用户继续引导到清单页或实操页；
- 不要独立成为孤岛。

说明卡片页适合做“认知整理”，不适合承担“真正学会”的职责。

## 教程文案规范

教程文案默认遵循下面这些规则。

### 规则一：文案必须面向动作

坏写法：

- “你可以在这里管理模型。”
- “这个页面支持高级能力配置。”

好写法：

- “先添加一个提供商，再启用至少一个模型。”
- “打开总开关后，模型才会看到这些工具。”
- “先在 iPhone 完成导入，再回手表确认是否同步成功。”

### 规则二：每屏只讲当前动作需要的信息

不要为了“完整”而塞很多背景说明。教程页的第一目标是帮助用户完成当前动作，而不是建立完整知识体系。

### 规则三：默认告诉用户下一步

一条教程如果没有明确下一步，用户很容易在完成当前动作后失去方向。

所以教程完成后必须明确告诉用户：

- 现在已经完成了什么；
- 接下来最推荐学什么；
- 如果不想继续，之后去哪里重新进入。

## ETOS 第一批教程应该先做什么

以后如果正式开始实现，默认第一批只做四条主线，不要一口气铺太大。

### 主线一：首次配置提供商

目标：

- 新用户成功添加一个提供商；
- 至少启用一个模型；
- 知道模型顺序和专用模型在哪里管理。

建议挂载位置：

- `ETOS LLM Studio/ETOS LLM Studio iOS App/Views/ProviderListView.swift`
- `ETOS LLM Studio/ETOS LLM Studio Watch App/Views/ProviderListView.swift`
- `ETOS LLM Studio/ETOS LLM Studio iOS App/Views/ProviderEditView.swift`
- `ETOS LLM Studio/ETOS LLM Studio Watch App/Views/ProviderEditView.swift`

推荐结构：

- 一个状态清单页；
- 一个最短路径说明卡；
- 必要时补一个字段解释页，解释 Base URL、API Key、API 格式分别是什么。

完成标准：

- 用户知道入口在哪；
- 用户成功保存一个提供商；
- 用户能确认至少一个模型可用；
- 用户能回到聊天页继续下一步。

### 主线二：第一次发起聊天

目标：

- 识别当前模型；
- 发送第一条消息；
- 看懂流式响应；
- 知道停止、继续、重新生成等基础动作。

建议挂载位置：

- `ETOS LLM Studio/ETOS LLM Studio iOS App/Views/ChatView.swift`
- `ETOS LLM Studio/ETOS LLM Studio Watch App/ContentView.swift`

推荐结构：

- 聊天空态中的最短路径入口；
- 2 到 3 条本地示例 Prompt；
- 发送成功后的下一步推荐。

完成标准：

- 用户发出第一条消息；
- 用户看到回复返回；
- 用户知道接下来如何创建新会话或切换设置。

### 主线三：工具箱上手

目标：

- 理解总开关与单项启用的关系；
- 知道导入、同步、暴露、调试分别在哪里；
- 理解 iPhone 和 watch 的职责分工。

建议挂载位置：

- `ETOS LLM Studio/ETOS LLM Studio iOS App/Views/MCPIntegrationView.swift`
- `ETOS LLM Studio/ETOS LLM Studio Watch App/Views/MCPIntegrationView.swift`
- `ETOS LLM Studio/ETOS LLM Studio iOS App/Views/ShortcutIntegrationView.swift`
- `ETOS LLM Studio/ETOS LLM Studio Watch App/Views/ShortcutIntegrationView.swift`
- `ETOS LLM Studio/ETOS LLM Studio iOS App/Views/ToolCenterView.swift`
- `ETOS LLM Studio/ETOS LLM Studio Watch App/Views/ToolCenterView.swift`

推荐结构：

- 继续保留现有说明卡片；
- 追加一个状态检查页；
- 把“已同步、已启用、已暴露、已验证”做成可见状态。

完成标准：

- 用户知道自己现在卡在哪一层；
- 用户能判断是导入问题、同步问题，还是工具暴露问题；
- 用户能完成一次最小验证闭环。

### 主线四：会话与 Daily Pulse

目标：

- 知道如何新建会话；
- 知道如何回到历史会话；
- 知道 Daily Pulse 和普通聊天的区别。

建议挂载位置：

- `ETOS LLM Studio/ETOS LLM Studio iOS App/Views/SessionListView.swift`
- `ETOS LLM Studio/ETOS LLM Studio Watch App/Views/SessionListView.swift`
- `ETOS LLM Studio/ETOS LLM Studio iOS App/Views/DailyPulseView.swift`
- `ETOS LLM Studio/ETOS LLM Studio Watch App/Views/DailyPulseView.swift`
- `ETOS LLM Studio/ETOS LLM Studio iOS App/Views/SettingsView.swift`
- `ETOS LLM Studio/ETOS LLM Studio Watch App/Views/SettingsView.swift`

推荐结构：

- 会话页的空态引导；
- Daily Pulse 的“第一次怎么用”入口；
- 完成后告诉用户它和普通聊天各适合什么场景。

完成标准：

- 用户能创建或切换会话；
- 用户能找到历史记录；
- 用户不会把 Daily Pulse 和普通会话混为一谈。

## 默认实现顺序

以后如果要正式开工，默认按这个顺序推进。

### 第一步：先补统一入口

先在 ETOS 里放一个长期可见的教程入口，不然所有后续内容都会变成散落的碎片。

### 第二步：先做提供商主线

因为如果连模型都没配通，其他教程几乎都没有意义。

### 第三步：再做首次聊天主线

这是用户最容易获得“第一次成功体验”的地方。

### 第四步：再做工具箱主线

工具箱更复杂，应该放在基础聊天闭环之后。

### 第五步：最后扩展到会话管理、Daily Pulse 和进阶主题

这些内容重要，但不是最短上手路径的第一优先级。

## 工程实现边界

教程功能默认遵守下面这些工程约束。

### 1. 不要在 SwiftUI `body` 里做重活

教程状态、样例内容、完成判定，默认都应该预计算。

严禁在 UI 渲染链路里同步执行：

- 磁盘 I/O；
- JSON 解码；
- 网络请求；
- 大量字符串拼装；
- 富文本重解析；
- 其他明显耗时操作。

教程系统本质上是辅助层，不能为了讲解把主界面拖慢。

### 2. 默认优先本地样例与临时状态

只要不是“配置真实服务本身”这一类教程，就优先使用：

- 本地示例 Prompt；
- 本地说明数据；
- 临时演示会话；
- 只读样例条目；
- 不落盘或可安全重置的练习状态。

### 3. 不要污染用户真实数据

教程不应该偷偷：

- 改写真实会话；
- 覆盖真实设置；
- 自动开启用户未确认的高风险能力；
- 留下用户难以理解的练习垃圾数据。

如果确实需要写入真实配置，必须让用户明确知道自己在做什么。

### 4. watchOS 必须比 iOS 更短、更浅

watchOS 教程默认遵守：

- 一屏一件事；
- 文案短；
- 层级浅；
- 页面可快速返回；
- 尽量减少大段说明。

iPhone 上可展开讲的内容，不要原封不动搬到手表上。

### 5. 不要做强制遮罩式教学

ETOS 不是只能给新人用的单功能 App。默认不要用那种“必须一步一步点完才让继续”的阻断式教程。

只在下面这些情况允许强提醒：

- 首次关键配置；
- 数据迁移；
- 不做就会导致功能完全不可用的前置条件。

其余情况都应以可跳过、可回看、可重入为原则。

## 建议的最小实现模型

真正开始实现时，第一版不要上来就做重框架。默认采用最小可维护结构即可。

建议至少有下面这些概念：

- 教程主线 ID；
- 教程步骤 ID；
- 页面类型；
- 是否已查看；
- 是否已完成；
- 最近完成时间；
- 当前入口来源；
- 支持的平台。

只要这几个状态能稳定表达，第一版就够用了。

如果后续确实需要跨 iOS 和 watchOS 共用进度，再把它收进共享层统一管理；不要在一开始就为了“未来可能扩展”做过度抽象。

## 验收标准

以后每加一条教程，至少拿下面这份清单自检一次。

### 必须全部满足

- 用户不看外部文档也能完成当前目标；
- 教程页只围绕一个核心动作；
- 用户做完后立刻得到成功反馈；
- 用户以后能重新找到这条教程；
- 教程不会污染真实数据；
- 教程不会显著拖慢界面；
- iOS 和 watchOS 的实现都符合各自平台习惯；
- 教程结束后明确告诉用户下一步是什么。

### 只要有一条不满足，就不算合格

- 需要再去读别的仓库代码才看得懂；
- 文案过多但动作路径不清晰；
- 页面里同时教很多件事；
- 用户做完后不知道自己是否成功；
- 教程只能第一次看，以后回不来；
- 教程和真实业务数据纠缠不清。

## 以后默认怎么执行

如果未来要让我直接为 ETOS 实现一条新手教程，默认按下面顺序处理：

1. 先确认它属于哪条主线；
2. 再确认它应该是状态清单页、实操练习页，还是说明卡片页；
3. 优先实现能形成最小闭环的版本；
4. 默认 iOS 和 watchOS 同时设计，但分别适配；
5. 优先挂在已有页面入口上，不凭空造新入口；
6. 只有当某条主线确定会长期扩展时，才抽更通用的数据模型。

## 最终原则

ETOS 的好教程，不是“写了很多说明”，而是让用户在最短路径里真正完成一次关键操作，并且以后随时能回来重学。

所以以后默认优先级永远是：

先让用户做成一次，再谈说明是否完整。
