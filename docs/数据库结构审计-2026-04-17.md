# ETOS LLM Studio 数据库结构超级详细审计（2026-04-17）

> 审计时间：2026-04-17  
> 审计方式：源码静态审计（未直接读取你设备上的实时 SQLite 文件）  
> 审计范围：`Shared/Shared/Persistence.swift`、`Shared/Shared/PersistenceGRDBStore.swift`、`Shared/Shared/Memory/*`、`Shared/Shared/SimilaritySearch/SQLiteVectorStore.swift`、`Shared/Shared/{ConfigLoader,WorldbookStore,MCPServerStore,ShortcutToolStore,FeedbackStore,ConversationMemoryManager}.swift`、iOS/watchOS 启动入口

---

## 0. 一句话结论

你当前数据库体系是**4库分治 + 启动期健康恢复 + 渐进迁移 + 关系表/JSON混合落盘**，总体工程化程度很高；相比旧版临时报告，现在可以明确确认：

- 配置库迁移已进入 **v5（枚举约束强化）**；
- 记忆库已稳定拆出 `memory-store.sqlite`，并配套向量库 `memory_vectors.sqlite`；
- 聊天主库具备 FTS、触发器、自动增量清理与启动恢复机制；
- 仍存在若干可继续优化点（明文密钥、部分弱约束、向量库写策略等）。

## 2. 存储拓扑总览（物理文件）

## 2.1 四个 SQLite 文件

| 类别 | 文件路径（Documents 下） | 主要职责 |
|---|---|---|
| 聊天主库 | `ChatSessions/chat-store.sqlite` | 会话、消息、文件夹、请求日志、Daily Pulse blob、会话摘要、FTS |
| 配置分库 | `Config/config-store.sqlite` | MCP、Provider、Worldbook、Shortcut、Feedback、配置类 legacy blob |
| 记忆分库 | `Memory/memory-store.sqlite` | 记忆原文（含 embedding_data）、用户画像、记忆类 legacy blob |
| 向量索引库 | `Memory/memory_vectors.sqlite` | 记忆分块向量索引（`memory_chunks`） |

> 说明：每个库都可能伴随 `-wal` / `-shm` 文件。

## 2.2 启动备份副本

启用 `sync.backup.createOnLaunch = true` 时，会在各库目录创建：

- `StartupBackups/chat-store.sqlite`
- `StartupBackups/config-store.sqlite`
- `StartupBackups/memory-store.sqlite`

特殊点：聊天备份会主动剥离 `messages_fts` 与 FTS 触发器，再 `VACUUM`，恢复后再由启动流程重建 FTS。

## 2.3 启动阶段时序

1. `AppLaunchStateMachine` 进入 `.preparingPersistence`；
2. 后台调用 `Persistence.bootstrapGRDBStoreOnLaunch()`；
3. 执行数据库健康检查与自动恢复（必要时从 `StartupBackups` 回灌）；
4. 初始化主库 + 配置库 + 记忆库；
5. 如聊天库被恢复，触发 `rebuildMessagesFTSIndex()`；
6. 按开关创建新的启动备份点；
7. 状态机进入 `.warmingServices` -> `.ready`。

## 3. 连接参数与维护策略

## 3.1 聊天主库（GRDB DatabasePool）

PRAGMA：

- `foreign_keys=ON`
- `journal_mode=WAL`
- `synchronous=NORMAL`
- `busy_timeout=5000`
- `wal_autocheckpoint=1000`
- `temp_store=MEMORY`
- `mmap_size=134217728`（128MB）

维护策略：

- 延迟维护（iOS/macOS 约 8s，watchOS 约 30s）；
- 若 `auto_vacuum != INCREMENTAL`，先切换并 `VACUUM` 一次；
- 触发增量回收阈值：`freelist >= 8192` 或空闲比例 `>= 0.2`；
- 单次回收上限：4096 页。

## 3.2 辅助分库（config/memory）

PRAGMA 与主库一致，区别：

- `mmap_size=67108864`（64MB）
- 增量回收阈值更保守：
  - `freelist >= 1024` 或空闲比例 `>= 0.25`
  - 单次回收上限 512 页

## 3.3 向量库（SQLite3 直连）

- 使用 `sqlite3_open` 直连，不经 GRDB；
- 写入流程：`BEGIN EXCLUSIVE TRANSACTION` -> `DELETE` 全表 -> 批量 `INSERT` -> `COMMIT`；
- 代码中未显式设置 WAL/synchronous/mmap 参数（沿用 SQLite 默认）。

## 4.3 全文检索对象（FTS5）

### 4.3.1 虚表 `messages_fts`

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS messages_fts
USING fts5(
  message_id UNINDEXED,
  session_id UNINDEXED,
  content,
  tokenize = 'unicode61'
)
```

### 4.3.2 同步触发器

- `messages_ai`：INSERT 同步入 FTS
- `messages_ad`：DELETE 时删 FTS 行
- `messages_au`：UPDATE 时删旧插新

### 4.3.3 备份与恢复特性

- 启动备份中会移除 `messages_fts` 和触发器；
- 恢复后调用 `rebuildMessagesFTSIndex()` 全量重建。

## 4.4 会话摘要策略

- 摘要字段位于 `sessions` 表内，不再单独建表；
- 提供：按会话 upsert / 单会话清空 / 全量清空 / 批量按更新时间读取；
- 更新时间排序使用 `COALESCE(conversation_summary_updated_at, updated_at)`。

## 4.5 启动期旧 JSON 导入策略（聊天域）

- 支持“全量导入 + 分阶段状态位 + 可选清理”；
- 若检测到不完整导入，可进入修复性重导入分支；
- 清理前会做导入校验，避免误删旧数据；
- 附带迁移对象：会话文件夹、请求日志、Daily Pulse blob。

- `MemoryRawStore.saveMemories` 会维护 `Memory/memories.json` 镜像；
- `ConversationUserProfileStore` 在 SQLite 成功后会删除 `user_profile.json`，失败再回退文件；
- 这是一种迁移期稳态策略：数据主源趋向 SQLite，但保留文件兼容通道。

## 7. 向量索引库 `memory_vectors.sqlite` 深度审计

## 7.1 表结构

### `memory_chunks`

- `chunk_id TEXT PRIMARY KEY`
- `parent_memory_id TEXT NOT NULL`
- `text TEXT NOT NULL`
- `embedding BLOB NOT NULL`
- `metadata TEXT NOT NULL`

索引：`idx_parent_memory(parent_memory_id)`

## 7.2 写入流程

1. 打开 `memory_vectors.sqlite`；
2. 建表/建索引（如不存在）；
3. `BEGIN EXCLUSIVE TRANSACTION`；
4. `DELETE FROM memory_chunks`（全量替换模式）；
5. 批量插入所有 chunk；
6. `COMMIT`。

## 7.3 与 `memory_items` 的关联方式

- 并非 FK 关联；
- 通过 `metadata["parentMemoryId"]` 软关联父记忆；
- 若缺失则回退用 `chunk_id` 本身。

## 10. 安全与可靠性审计结论

## 10.1 高优先级（建议尽快）

1. **敏感信息明文落盘**
   - `providers.api_key`（子表）、`providers.proxy_password`、`mcp_servers.api_key` 等为明文字段。
   - 建议：至少做 Keychain/本地加密封装，DB 只存引用或密文。

2. **向量库全量重写策略在大数据量下会放大写放大与锁占用**
   - 当前每次 `DELETE + 全量 INSERT`。
   - 建议：改增量 upsert（按 chunk_id diff），并设置 WAL + 合理 checkpoint。

## 10.2 中优先级（可排期）

3. **部分枚举字段缺少 DB CHECK**
   - `request_logs.status`、`provider_models.request_body_override_mode`、`shortcut_tools.run_mode_hint`、`worldbook_entry_keys.key_kind`。

4. **部分外键未下沉到 DB**
   - `sessions.folder_id`、`request_logs.session_id/provider_id`、`conversation_user_profile.source_session_id`。

5. **FTS 目前主要是写侧维护，读侧未见明确统一查询入口**
   - 若长期未使用，存在额外写放大与备份复杂度；
   - 建议明确：要么接入搜索功能，要么降级为按需构建。

## 10.3 低优先级（持续优化）

6. **JSON 镜像与 SQLite 双轨在极端失败场景可能出现短时漂移**
   - 目前已通过“成功后清理/回退补写”尽量减轻。

7. **索引可继续按真实查询分布复核**
   - 现有索引覆盖已经不错，但可用慢查询日志继续精简。
